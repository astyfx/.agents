## `.agents` 개편 프로젝트 기획서

### 1. 문제 정의

현재 `.agents`는 정책, skill, subagent, eval 문서는 잘 갖춰졌지만, 실제로는 “작업 수행 시스템”보다 “잘 정리된 설정 저장소”에 더 가깝습니다.

- `tracking`은 구조는 있으나 실운용 가치는 약합니다. 최근에도 [TRACKING.md](/Users/jacob.kim/.agents/docs/instructions/TRACKING.md), [work-handoff.md](/Users/jacob.kim/.agents/work-handoff.md) 중심으로 가벼워졌지만, 결국 대부분의 상태는 `work-handoff.md` 하나에 몰립니다.
- `learnings`는 “자연 증식” 구조가 아니라 수동 문서 모음입니다. [learnings/README.md](/Users/jacob.kim/.agents/learnings/README.md)는 기준은 좋지만, 작업 종료 후 학습이 자동으로 증류되는 루프가 없습니다.
- 현재 [ROADMAP.md](/Users/jacob.kim/.agents/ROADMAP.md)는 모든 phase 완료로 표시돼 있지만, 실제로는 “성장 메커니즘”보다 “기능 나열”에 더 가깝습니다.
- 메모리도 최소 포인터 수준입니다. [MEMORY.md](/Users/jacob.kim/.agents/claude/projects/-Users-jacob-kim--agents/memory/MEMORY.md), [project_harness_evolution.md](/Users/jacob.kim/.agents/claude/projects/-Users-jacob-kim--agents/memory/project_harness_evolution.md)만으로는 장기 축적이 일어나지 않습니다.
- 가장 큰 구조적 문제는 `정책`, `작업 handoff`, `장기 지식`, `문제 해결 기록`, `평가`가 서로 다른 수명주기를 가져야 하는데 지금은 경계가 흐립니다.

### 2. 다른 harness 구조의 가치평가

- 문서 중심 policy hub:
  - 장점: Claude/Codex parity, 규칙 통제, 이식성.
  - 단점: 토큰 비용 증가, 문서 drift, 학습 정체.
- wiki-first knowledge harness:
  - 장점: 지식 누적, 비교/종합/회고에 강함.
  - 단점: 실제 실행 절차와 handoff 규율이 약하면 “좋은 위키”로만 끝남.
- plan-first harness:
  - 장점: 큰 작업, 리팩터링, 장기 멀티세션 작업에 강함.
  - 단점: 모든 작업에 강제하면 과합니다.
- eval-first ops harness:
  - 장점: 개선이 측정 가능해짐.
  - 단점: 계측 없이는 형식만 남습니다.

권장 방향은 하나를 고르는 게 아니라, `kernel + playbooks + operational memory + knowledge wiki + eval loop`의 하이브리드입니다.

### 3. 목표 상태

`.agents`를 “공유 규칙 저장소”가 아니라 아래 역할을 가진 운영체계로 재정의합니다.

- 최소한의 공통 프롬프트 커널만 항상 로드한다.
- 필요한 지식과 절차는 skill/playbook으로 지연 로드한다.
- 단기 상태와 장기 지식을 분리한다.
- Claude와 Codex가 같은 문서 체계와 artifact를 읽고 이어받을 수 있게 한다.
- 작업 중 반복되는 실패를 `troubleshooting` 자산으로 축적한다.
- 팀 레포에는 개인화 흔적이 남지 않게 하고, 개인 자산은 글로벌 harness에 남긴다.
- MCP/Slack/Jira/GitHub/Figma 같은 외부 통합을 “도구 연결”이 아니라 “업무 루프”로 묶는다.

### 4. 제안 아키텍처

```text
.agents/
  core/                 # 항상 로드되는 최소 규칙
  playbooks/            # skills, subagents, reusable workflows
  knowledge/            # 장기 지식
  knowledge/wiki/       # 개념, 패턴, 비교, 운영 원칙
  knowledge/troubleshooting/  # 반복 실수, 장애, 디버깅 기록
  knowledge/decisions/  # ADR, 정책 변경 이유
  memory/               # 현재 active focus와 사용자/역할 컨텍스트
  plans/                # 큰 작업용 ExecPlan/PRD/Refactor plan
  ops/                  # eval, metrics, weekly review, health checks
  integrations/         # MCP/integration 운영 규칙
  repo-init/            # 팀 친화적 per-repo scaffold
```

핵심 원칙은 다음입니다.

- `AGENTS.md`는 얇게 유지합니다.
- 기본 로드 문서는 `core/`만 남깁니다.
- `tracking/`은 기본 구조에서 내리고, 큰 작업에서만 `plans/` 또는 `execplan`로 승격합니다.
- `work-handoff.md`는 단기 실행 상태로만 씁니다.
- `knowledge/`는 사람이 읽는 위키이면서 agent가 ingest/query/lint 가능한 구조로 만듭니다.

### 5. 개편안

- `tracking` 기본값 폐지:
  - 일반 작업은 `work-handoff.md`만 사용.
  - 멀티세션, 고위험, 리팩터링만 `plans/<topic>.md` 사용.
- `learnings`를 `knowledge`로 승격:
  - `learnings/`는 `knowledge/wiki/engineering/`로 흡수.
  - 별도로 `knowledge/troubleshooting/`, `knowledge/decisions/`, `knowledge/workflows/` 추가.
- `memory` 재설계:
  - 현재의 user/project 메모는 유지하되, “현재 왜 이런 구조를 쓰는지”와 “지금 active redesign thesis가 무엇인지”를 별도 문서로 유지.
- `ROADMAP` 재정의:
  - “phase complete” 문서가 아니라, 실측 지표와 개편 backlog를 가진 살아있는 운영 로드맵으로 전환.
- `skills` 재분류:
  - 실행 스킬, 분석 스킬, 운영 스킬, 통합 스킬로 묶고, trigger를 더 좁혀 토큰 낭비를 줄임.
- Claude/Codex handoff 표준화:
  - 둘 다 동일한 `work-handoff.md`, `plans/*.md`, `knowledge/*`를 읽게 함.
  - 모델별 차이는 bridge와 runtime config에만 남김.
- team-safe per-repo 전략 강화:
  - per-repo에는 얇은 안내 파일만 커밋.
  - 개인 메모, handoff, local overrides, workspace state는 반드시 글로벌 또는 gitignored local에 둠.

### 6. 실행 단계

- Phase A: Baseline reset
  - [ROADMAP.md](/Users/jacob.kim/.agents/ROADMAP.md), [ARCHITECTURE.md](/Users/jacob.kim/.agents/ARCHITECTURE.md), [CHANGELOG.md](/Users/jacob.kim/.agents/CHANGELOG.md)를 “현재 구조 설명”과 “목표 구조”로 분리.
- Phase B: Prompt kernel slimming
  - [AGENTS.md](/Users/jacob.kim/.agents/AGENTS.md)를 더 줄이고, 큰 설명은 `core/`와 `playbooks/`로 이동.
- Phase C: Knowledge system
  - `knowledge/wiki`, `knowledge/troubleshooting`, `knowledge/decisions` 신설.
  - 작업 종료 시 1회성 요약이 아니라 ingest 규칙과 lint 규칙을 도입.
- Phase D: ExecPlan adoption
  - 큰 작업만 `plans/` 기반으로 운영.
  - 현재 `tracking`은 deprecate 후 마이그레이션.
- Phase E: Growth loop
  - 주간 review: 반복 실수, 자주 쓰는 MCP, 실패한 handoff, 토큰 큰 규칙 재정리.
  - `evals`를 실제 구조 개선 판단에 연결.
- Phase F: Integration ops
  - Slack/Jira/GitHub/Figma/MCP별로 “어떻게 호출하나”가 아니라 “어떤 업무 루프에서 언제 쓰나” 문서화.

### 7. 성공 기준

- 기본 프롬프트 로드 토큰이 지금보다 의미 있게 감소한다.
- Claude → Codex, Codex → Claude handoff 성공률이 올라간다.
- 동일한 실수 유형이 `knowledge/troubleshooting` 축적 후 반복 감소한다.
- 큰 작업은 `plans/`만으로 재개 가능하고, 작은 작업은 별도 tracking 없이도 이어진다.
- 팀 레포에 개인 운영 흔적이 남지 않는다.
- roadmap가 “완료 체크리스트”가 아니라 “지속 개편 운영판”으로 작동한다.

## 판단

이번 개편의 핵심은 `tracking을 고치는 것`이 아니라, `.agents`를 `정책 저장소`에서 `학습하는 운영체계`로 바꾸는 것입니다.  
즉, 기본은 가볍게, 큰 일만 깊게, 지식은 누적 가능하게, handoff는 모델 중립적으로, 개선은 측정 가능하게 가져가야 합니다.
