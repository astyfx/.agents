# Plan

## Scope

Build a forward-looking plan for evolving this repository into a stronger personal AI agent harness by combining:
- analysis of the current repo structure and recent changes
- inference about the user's current AI usage style from repository artifacts
- up-to-date external references from official docs, GitHub repos/issues, and selected public discussions

## Assumptions

- The repo is the current source of truth for the user's harness direction.
- The user values cross-agent portability across Claude and Codex more than deep optimization for only one tool.
- The user prefers minimal, human-written policy files over generated boilerplate.
- The immediate deliverable is a strategic roadmap, not code implementation.

## Current Repo Diagnosis

### Strengths already present

- Centralized shared policy in `AGENTS.md`
- Runtime indirection and bridge files for Claude/Codex
- Skill-oriented architecture with reusable `SKILL.md` folders
- Explicit engineering conventions, library choices, and tracking rules
- Strong preference for reproducibility and versioned configuration
- Intentional separation of global policy vs runtime state

### Likely user operating style inferred from the repo

- Uses AI as a daily engineering copilot across multiple repositories, not as a one-off chat tool
- Wants persistent behavior across sessions and across agents
- Cares about controllability more than novelty
- Prefers short human-authored rules over opaque autogenerated context
- Treats skills as reusable operating procedures
- Wants the harness itself to become a compounding asset

## Done Criteria

- Produce a repo-specific roadmap with clear priorities, phases, and rationale
- Ground recommendations in both local evidence and external sources
- Identify what should be preserved, not only what should be added
- Recommend a sequence that improves reliability, observability, and leverage without bloating the harness

## Roadmap

### Preserve as non-negotiable

1. Keep `AGENTS.md` minimal and human-written.
2. Keep runtime bridges thin.
3. Keep skills as modular directories with local assets/scripts.
4. Keep tracking artifacts for substantial work.

### Phase 1: Add observability before adding more intelligence

Goal:
Make agent behavior measurable so future harness changes are evidence-driven.

Work:
- Add a lightweight `evals/` or `benchmarks/` area with representative tasks
- Define success/failure taxonomy for agent runs
- Persist session summaries, skill usage, failures, retries, and recovery patterns in structured form
- Create a comparison workflow for Claude vs Codex on the same task set

Success metrics:
- time-to-first-correct-change
- number of user corrections
- verification pass rate
- token or cost proxy if available
- retry count and failure category

### Phase 2: Move hard guarantees from prompts into deterministic mechanisms

Goal:
Reduce reliance on static instruction files for rules that must always hold.

Work:
- Use hooks or wrapper scripts for protected files, commit validation, secret protection, and routing guardrails
- Keep `AGENTS.md` for minimal policy and preference only
- Introduce path-aware or task-aware context injection for high-value cases instead of global context bloat

Success metrics:
- fewer policy violations
- lower instruction drift between agents
- smaller base policy surface

### Phase 3: Turn skills into a real portability layer

Goal:
Make skills the main reusable unit across tools, not just Claude-local convenience files.

Work:
- Add manifest metadata or lintable structure for skills
- Split heavy skills into overview + references + scripts more consistently
- Classify skills by function: research, planning, review, implementation, UI, release
- Add test prompts or sanity checks per skill
- Design export paths toward plugin/marketplace formats where useful

Success metrics:
- faster skill authoring
- fewer undertrigger/overtrigger cases
- reusable skills across Claude/Codex/Gemini-style ecosystems

### Phase 4: Add orchestration and memory intentionally

Goal:
Upgrade from single-agent customization to controlled multi-agent workflows.

Work:
- Define a small set of explicit subagent roles
- Add a task router: when to stay local, when to research, when to review, when to delegate
- Build compact memory artifacts from completed tasks: pitfalls, project heuristics, preferred commands
- Avoid generic long-term memory dumps; prefer structured, scoped memory

Success metrics:
- less context loss on long tasks
- better decomposition quality
- less duplicated analysis work

### Phase 5: Package the harness as a product for yourself

Goal:
Treat this repo like an internal platform, not a folder of prompt files.

Work:
- Add a top-level architecture doc for the harness itself
- Add a changelog or release notes for harness changes
- Add bootstrap validation commands and self-check scripts
- Consider a Claude plugin marketplace layout for team/private distribution
- Consider an interoperability layer for portable skills and agent configs

Success metrics:
- reproducible setup on a new machine
- lower breakage after upgrades
- easier comparison across agent vendors

## Priority Order

1. Observability and evals
2. Deterministic guardrails via hooks/wrappers
3. Skill portability and testing
4. Explicit orchestration and scoped memory
5. Packaging/distribution

## Why this order

- The repo is already strong on policy structure.
- The biggest missing layer is measurement.
- Without evals, new hooks/skills/memory systems will feel useful but be hard to validate.
- External evidence suggests oversized context files can hurt performance and cost, so the next step should be dynamic mechanisms and measurement, not more static prose.

## Immediate 30-Day Plan

### Week 1

- Add `benchmarks/` with 10 to 20 representative tasks taken from real use:
  - repo policy refactor
  - multi-file code edit
  - review-only bug finding
  - prompt refinement
  - skill creation
- Add a scorecard with pass/fail, rework count, verification quality, and human time saved.

### Week 2

- Add `scripts/new-task.sh` to scaffold required tracking files automatically.
- Add `scripts/check-harness.sh` to validate:
  - required policy files exist
  - links to docs are not broken
  - tracking folders include required files
  - skills have valid frontmatter and expected layout

### Week 3

- Introduce a `skills/schema.md` or JSON schema for skill metadata.
- Add per-skill sanity tests or canonical prompts.
- Split larger skills into `SKILL.md` plus `references/`, `assets/`, or `scripts/` where appropriate.

### Week 4

- Add the first reusable subagents:
  - researcher
  - reviewer
  - implementer
  - release/checker
- Add a short orchestration policy for when to stay single-agent versus when to delegate.
